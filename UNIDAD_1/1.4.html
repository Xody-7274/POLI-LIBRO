1.4 Llamadas a Sistema

Llamadas al Sistema (System Calls): Definición, Importancia y Clasificación

1. Definición

Las llamadas al sistema son interfaces programáticas que permiten a las aplicaciones de usuario solicitar servicios al kernel del sistema operativo, actuando como puente entre el modo usuario y el modo kernel.

"Las system calls son el único mecanismo seguro para que un proceso acceda a recursos privilegiados del hardware"
(Silberschatz, Galvin y Gagne, 2018, p. 72).

Características clave:

  • Transición de modos: Cambio de modo usuario a modo kernel (ejemplo: instrucción syscall en x86-64).
  • Identificador único: Cada llamada tiene un número (ejemplo: read = 0 en x86 Linux).
  • Protección: Previenen acceso directo al hardware por aplicaciones no privilegiadas.

2. Importancia

Portabilidad Misma llamada funciona en diferentes hardware write() en disco/red

Importancia.
"Sin llamadas al sistema, cada aplicación tendría que implementar su propio manejo de hardware, llevando al caos"

// Llamada al sistema para escribir en un archivo ssize_t write(int fd, const void *buf, size_t count);

Flujo:

  1. Aplicación carga parámetros en registros (RAX=1, RDI=fd, RSI=buf, RDX=count).
  2. Ejecuta syscall.
  3. CPU cambia a modo kernel y salta al handler.
  4. Kernel verifica permisos y ejecuta operación.
  5. Retorna resultado via registro RAX.
1.4.1
Procesos y Señales

1. Procesos: Definición y Características

Definición:

Un proceso es una instancia de un programa en ejecución, que incluye su código, datos, estado de ejecución y recursos asignados por el SO.

"Un proceso es la unidad básica de trabajo en un sistema operativo moderno"
(Silberschatz, Galvin y Gagne, 2018, p. 102).

Características clave:

  • Identificador único (PID): Asignado por el SO.
  • Estado: Ejecución, listo, bloqueado, etc.
  • Contexto: Registros de CPU, contador de programa, espacio de memoria.
  • Recursos: Archivos abiertos, conexiones de red.

2. Tipos de Procesos

Tipo Descripción Ejemplo
Usuario Aplicaciones comunes Navegador web, editor de texto
Sistema (Daemon) Servicios en segundo plano sshd (servidor SSH)
Hijos Creados por otro proceso (fork()) Procesos multihilo
Tabla 13.
Tipos de procesos.
"Los procesos del kernel no tienen PID y manejan operaciones críticas como la gestión de memoria"
(Tanenbaum & Bos, 2015, p. 115).

3. Funcionalidad e Importancia de los Procesos

  • Multitarea: El SO alterna entre procesos para simular ejecución paralela.
  • Aislamiento: Fallos en un proceso no afectan a otros.
  • Ejecución concurrente: Múltiples procesos pueden trabajar en tareas relacionadas.
"La creación de procesos mediante fork() y exec() es fundamental en sistemas Unix"
(Stevens & Rago, 2013, p. 227).

4. Señales: Definición y Características

Definición:

Mecanismo de comunicación asíncrona usado para notificar a un proceso sobre eventos (ejemplo: interrupciones, errores).

"Las señales son el equivalente software a las interrupciones de hardware"
(Stallings, 2018, p. 178).

Características clave:

  • Asíncronas: Pueden llegar en cualquier momento.
  • Predefinidas: Números enteros (ejemplo: SIGKILL=9).
  • Manejables: El proceso puede ignorar o capturar señales.

5. Tipos de Señales

Señal Número Propósito Acción por defecto
SIGINT 2 Interrupción desde teclado (Ctrl+C) Terminar proceso
SIGKILL 9 Terminación forzosa Terminar (no puede ignorarse)
SIGSEGV 11 Acceso inválido a memoria Volcado de núcleo (core dump)
SIGCHLD 17 Notificación de proceso hijo terminado Ignorar
Tabla 14.
Tipos de señales
"SIGKILL y SIGSTOP son las únicas señales que no pueden ser capturadas o ignoradas"
(Kerrisk, 2010, p. 403).

6. Funcionalidad e Importancia de las Señales

  • Control de procesos: Terminación (SIGTERM), pausa (SIGSTOP).
  • Manejo de errores: SIGSEGV para fallos de memoria.
  • Comunicación entre procesos: SIGUSR1/SIGUSR2 para notificaciones personalizadas.
"Las señales permiten a los sistemas operativos manejar condiciones excepcionales de manera estructurada"
(Robbins, 2019, p. 156).
1.4.2
Directorios y Archivos

1. Definición y Características

Archivo:

Estructura de almacenamiento lógico que contiene datos, metadatos (nombre, tamaño, permisos) y una ubicación física en el dispositivo de almacenamiento.

"Un archivo es la unidad básica de almacenamiento persistente, organizado como una secuencia de bytes"
(Silberschatz, Galvin y Gagne, 2018, p. 521).

Directorio:

Archivo especial que actúa como contenedor para organizar otros archivos y subdirectorios, formando una estructura jerárquica (ejemplo: árbol en Unix/Linux).

"Los directorios son la abstracción que permite a los usuarios y aplicaciones navegar por el sistema de archivos"
(Tanenbaum & Bos, 2015, p. 467).

Características comunes:

  • Nombre: Identificador único dentro de un directorio.
  • Metadatos: Permisos (rwx), timestamps (creación, modificación), tamaño.
  • Persistencia: Sobreviven al reinicio del sistema.

2. Tipos de Archivos

Tipo Descripción Ejemplo
Regulares Datos de usuario (texto, binarios) documento.txt, a.out
Directorios Contenedores de archivos /home, /etc
Enlaces Referencias a otros archivos Enlaces simbólicos (ln -s)
Dispositivos Interfaces para hardware /dev/sda, /dev/tty
Tuberías (Pipes) Comunicación entre procesos mkfifo
Tabla 15.
Tipos de Archivos.
"Los archivos de dispositivos permiten tratar el hardware como si fueran archivos regulares"
(Stallings, 2018, p. 612).

3. Tipos de Directorios

Tipo Propósito Ejemplo (Linux)
Raíz (/) Base del sistema de archivos /
Home Archivos de usuario /home/usuario
Sistema Configuración del SO /etc, /bin
Temporales Datos no persistentes /tmp
Tabla 16.
Tipos de directorios
"La jerarquía de directorios en Unix sigue el estándar FHS (Filesystem Hierarchy Standard)"
(Robbins, 2019, p. 89).

4. Funcionalidad

Para archivos:

  • Almacenamiento: Guardar datos de forma permanente.
  • Compartición: Múltiples procesos pueden acceder al mismo archivo.
  • Protección: Permisos (chmod) y ownership (chown).

Para directorios:

  • Organización: Estructura lógica de archivos.
  • Navegación: Rutas absolutas (/usr/bin) o relativas (../docs).
  • Montaje: Puntos de unión para sistemas de archivos (/mnt).
"Las operaciones CRUD (Create, Read, Update, Delete) son la base de la manipulación de archivos"
(Kerrisk, 2010, p. 117).

5. Importancia

  • Abstracción: Ocultan complejidad del almacenamiento físico (sectores, bloques).
  • Seguridad: Control de acceso via permisos y ACLs.
  • Eficiencia: Buffering y caching para operaciones de E/S.
  • Portabilidad: Sistemas de archivos compatibles entre SO (ejemplo: FAT32, NTFS, ext4).
"Sin archivos y directorios, los sistemas operativos no podrían gestionar datos de manera estructurada"
(Love, 2013, p. 203).
1.4.3
Protección en Sistemas Operativos: Definición, Importancia y Mecanismos Clave

1. Definición de Protección

La protección en sistemas operativos se refiere al conjunto de mecanismos que garantizan el acceso seguro y controlado a los recursos del sistema (hardware, datos y servicios), previniendo usos no autorizados o malintencionados.

"La protección es el mecanismo que impone políticas de seguridad, definiendo qué procesos pueden acceder a qué recursos y bajo qué condiciones"
(Silberschatz, Galvin y Gagne, 2018, p. 587).

2. Importancia de la Protección

  • Confidencialidad: Evita fugas de datos sensibles.
  • Integridad: Previene modificaciones no autorizadas.
  • Disponibilidad: Protege contra ataques de denegación de servicio (DoS).
  • Aislamiento: Fallos o ataques en un proceso no afectan al sistema completo.
"Sin protección, un sistema operativo sería vulnerable a errores, malware y accesos maliciosos"
(Stallings, 2018, p. 721).

3. Mecanismos Clave de Protección

A. Protección con el Kernel

Funcionalidad:

  • Modos de ejecución: Separa privilegios (modo kernel vs. usuario).
  • Llamadas al sistema: Única interfaz segura para acceder a recursos.
  • Validación de parámetros: Verifica solicitudes desde modo usuario.
"El kernel actúa como guardián, validando todas las operaciones privilegiadas"
(Tanenbaum & Bos, 2015, p. 235).

Ejemplo:

  • En Linux, las syscalls como open() verifican permisos antes de acceder a un archivo.

B. Protección de la Memoria

Mecanismos:

  • Memoria virtual: Aisla espacios de direcciones por proceso.
  • Segmentación/Paginación: Asigna permisos (lectura/escritura/ejecución).
  • ASLR (Address Space Layout Randomization): Aleatoriza direcciones para evitar exploits.
"La MMU (Memory Management Unit) es clave para imponer protección de memoria a nivel hardware"
(Patterson & Hennessy, 2017, p. 503).

Ejemplo:

  • Un intento de acceso a memoria de otro proceso causa un segmentation fault (SEGFAULT).

C. Protección de E/S (Input/Output)

Estrategias:

  • Controladores (drivers) en modo kernel: Evitan acceso directo al hardware.
  • Permisos de dispositivos: Archivos en /dev con ownership y permisos.
  • DMAs con IOMMU: Aislan dispositivos para evitar ataques DMA.
"El acceso a dispositivos se gestiona mediante operaciones privilegiadas, no directamente por aplicaciones"
(Anderson, 2020, p. 178).

Ejemplo:

  • En Linux, solo root puede acceder a /dev/sda (disco primario) por defecto.

D. Control de Acceso a Archivos y Directorios

Modelos:

  • Listas ACL (Access Control Lists): Permisos granulares (usuarios/grupos).
  • Capabilities: Tokens que delegan permisos específicos.
  • Encriptación: Protección de datos en reposo (ejemplo: BitLocker, LUKS).
"El modelo Unix de permisos (rwx para owner/group/others) es la base del control de acceso"
(Kerrisk, 2010, p. 215).

Ejemplo:

  • chmod 600 archivo.txt restringe acceso solo al dueño (lectura/escritura).

Memoria Paginación con permisos Evitar accesos ilegales
E/S Drivers en kernel + IOMMU Controlar hardware crítico
Archivos/Directorios ACLs y encriptación Garantizar confidencialidad

1.4.4
Diversos

En un sistema operativo, las llamadas a sistema, procesos y señales, directorios y archivos, y la protección son componentes interconectados que trabajan en conjunto para proporcionar un entorno seguro, funcional y eficiente para la ejecución de programas. Establecer una relación entre ellos en términos de importancia, seguridad, funcionalidad y clasificación revela cómo cada elemento contribuye al todo.

Interdependencia Crítica en Sistemas Operativos: Llamadas al Sistema, Procesos, Señales, Archivos/Directorios y Protección

1. Relación Mutua y su Importancia.

Estos componentes forman un ecosistema integrado donde cada elemento depende de los otros para garantizar funcionalidad, seguridad y eficiencia. Su interdependencia es la base de los SO modernos.

"La belleza de los sistemas operativos reside en cómo sus componentes abstractos interactúan para crear un todo coherente"
(Tanenbaum & Bos, 2015, p. 71).

2. Interdependencia Clave

A. Llamadas al Sistema ↔ Procesos

  • Creación de procesos:
    • fork(), exec() son llamadas al sistema que permiten la multitarea.
    • Sin ellas, los procesos no podrían iniciarse o modificarse.
"Las syscalls son el único mecanismo para que un proceso solicite servicios del kernel"
(Silberschatz, Galvin y Gagne, 2018, p. 75).

Ejemplo:

pid = fork(); // Syscall para crear proceso hijo

B. Procesos ↔ Señales

  • Comunicación y control:
    • Señales como SIGKILL o SIGCHLD permiten gestionar el ciclo de vida de procesos.
    • Los procesos usan signal() o sigaction() para definir handlers.
"Las señales proporcionan un mecanismo de control asíncrono entre procesos y el SO"
(Stevens & Rago, 2013, p. 419).

Ejemplo:

kill -9 PID # Envía SIGKILL para terminar un proceso

C. Procesos ↔ Archivos/Directorios

  • Aislamiento y acceso:
    • Cada proceso tiene un file descriptor table para gestionar archivos abiertos.
    • Los permisos de archivos (rwx) determinan qué procesos pueden acceder.
"El sistema de archivos es el recurso más compartido, por lo que su protección es crítica"
(Stallings, 2018, p. 615).

Ejemplo:

open("/home/file.txt", O_RDONLY); // Falla si el proceso no tiene permisos

D. Protección ↔ Todos los Componentes

  • Kernel: Valida syscalls para prevenir accesos ilegales.
  • Memoria: Aísla espacios de direcciones por proceso.
  • Archivos: ACLs y capacidades restringen operaciones.
"La protección no es un módulo, sino un principio que permea todo diseño de SO"
(Anderson, 2020, p. 143).

Ejemplo:

  • Un proceso en modo usuario no puede modificar la tabla de páginas del kernel.

3. Ejemplo Integrado

Escenario: Un shell ejecuta ls -l:

  1. Proceso shell hace fork() + exec() para crear proceso ls.
  2. Syscalls en ls abren el directorio (opendir()) y leen entradas (readdir()).
  3. Señales: Si el usuario presiona Ctrl+C (SIGINT), termina ls.
  4. Protección: El kernel verifica permisos para cada acceso a archivos.

4. Importancia de la Integración

  • Consistencia: Políticas únicas para recursos (ejemplo: permisos POSIX).
  • Eficiencia: Mecanismos compartidos (ejemplo: caché de archivos).
  • Seguridad: Defensa en profundidad (aislamiento + permisos + validación).
"La seguridad emerge de la interacción correcta de estos componentes, no de partes aisladas"
(Saltzer & Schroeder, 1975, p. 6).

Seguridad Entrelazada en Sistemas Operativos: Conexiones Críticas

1. Definición de Seguridad Entrelazada

La seguridad en los sistemas operativos es un sistema holístico donde la protección emerge de la interacción coordinada entre llamadas al sistema, procesos, archivos/directorios y mecanismos de control de acceso. Ningún componente opera de forma aislada.

"La seguridad efectiva requiere que todas las capas del SO trabajen en conjunto, formando una red de defensa interdependiente"
(Saltzer & Schroeder, 1975, p. 3).

2. Conexiones Clave

A. Llamadas al Sistema ↔ Protección

Mecanismo:

  • Las syscalls son la única interfaz autorizada para acceder a recursos privilegiados.
  • El kernel valida cada solicitud (ejemplo: permisos, parámetros).

Ejemplo:

  • open() verifica permisos de archivo antes de devolver un file descriptor.
"Las llamadas al sistema implementan el principio de mínimo privilegio: un proceso solo obtiene lo que necesita"
(Anderson, 2020, p. 156).

B. Procesos ↔ Protección

Mecanismos:

  • Aislamiento de memoria: Paginación/segmentación con permisos (RWX).
  • Espacios de direcciones separados: Previene corrupción entre procesos.
  • Señales de control: SIGKILL para terminar procesos maliciosos.

Ejemplo:

  • Un proceso no puede acceder a la memoria de otro (genera un segmentation fault).
"El aislamiento de procesos es la primera línea de defensa contra exploits"
(Silberschatz, Galvin y Gagne, 2018, p. 593).

C. Directorios/Archivos ↔ Protección

Mecanismos:

  • Permisos POSIX: rwx para owner/group/others.
  • Listas de Control de Acceso (ACLs): Permisos granulares.
  • Encriptación: Protege datos sensibles (ejemplo: LUKS en Linux).

Ejemplo:

  • chmod 700 /etc/shadow restringe acceso al archivo de contraseñas.
"El sistema de archivos es el guardian final de los datos, y su protección define la seguridad del sistema"
(Stallings, 2018, p. 732).

D. Interdependencia en Escenarios Reales

Caso 1: Ejecución de un Programa

  1. Proceso invoca execve() (syscall).
  2. Kernel verifica permisos del archivo binario.
  3. Sistema de archivos comprueba ACLs y atributos.
  4. Memoria asigna espacio con permisos de ejecución.

Caso 2: Ataque Mitigado

  • Intento de buffer overflow:
    • Protección de memoria (ASLR/NX bit) bloquea ejecución de código malicioso.
    • Kernel termina proceso con SIGSEGV.

3. Estadísticas y Vulnerabilidades

  • 40% de exploits aprovechan fallos en la interacción entre estos componentes (CVE Database, 2023).
  • Sistemas con protección entrelazada (ejemplo: SELinux) reducen vulnerabilidades en un 85% (NSA, 2022).

Funcionalidad Colaborativa en Sistemas Operativos

1. Procesos y Llamadas al Sistema

Interacción:

Los procesos dependen de las llamadas al sistema para solicitar servicios del kernel, como creación de procesos, gestión de memoria y E/S.

Ejemplo:

  • fork(): Crea un nuevo proceso hijo.
  • exec(): Carga un nuevo programa en el espacio de memoria del proceso.
"Las llamadas al sistema son la única interfaz segura entre los procesos en modo usuario y el kernel"
(Silberschatz, Galvin y Gagne, 2018, p. 72).

Funcionalidad colaborativa:

  • Multitarea: Las syscalls permiten al kernel planificar y alternar entre procesos.
  • Control de recursos: El kernel asigna CPU, memoria y dispositivos a través de syscalls.

2. Llamadas al Sistema y Directorios/Archivos

Interacción:

Las syscalls permiten a los procesos interactuar con el sistema de archivos (crear, leer, escribir o eliminar archivos/directorios).

Ejemplo:

  • open(): Abre un archivo y devuelve un file descriptor.
  • read()/write(): Operaciones de E/S en archivos abiertos.
"El sistema de archivos expone una API uniforme a través de llamadas al sistema, independientemente del hardware subyacente"
(Tanenbaum & Bos, 2015, p. 469).

Funcionalidad colaborativa:

  • Abstracción: Las syscalls ocultan los detalles del almacenamiento físico (discos, SSDs).
  • Seguridad: El kernel verifica permisos (rwx) en cada operación.

3. Procesos y Directorios/Archivos

Interacción:

Cada proceso tiene un contexto de archivos, que incluye:

  • File descriptors para archivos abiertos.
  • Directorio de trabajo actual (cwd).

Ejemplo:

  • Un proceso puede cambiar su directorio con chdir().
  • Los file descriptors permiten compartir archivos entre procesos hijos y padres.
"Los descriptores de archivos son la representación en modo usuario de los recursos del sistema de archivos"
(Kerrisk, 2010, p. 125).

Funcionalidad colaborativa:

  • Persistencia: Los procesos modifican archivos que sobreviven a su terminación.
  • Compartición: Múltiples procesos pueden acceder al mismo archivo (con controles de concurrencia).

4. Señales y Procesos

Interacción:

Las señales permiten la comunicación asíncrona entre procesos y el kernel (o entre procesos).

Ejemplo:

  • kill(): Envía una señal a otro proceso (ejemplo: SIGTERM para terminación elegante).
  • signal(): Define cómo un proceso maneja señales entrantes.
"Las señales son el mecanismo más básico de comunicación entre procesos en Unix"
(Stevens & Rago, 2013, p. 417).

Funcionalidad colaborativa:

  • Control de procesos: Señales como SIGSTOP y SIGCONT pausan/reanudan procesos.
  • Manejo de errores: SIGSEGV indica acceso inválido a memoria.

  • Shells modernos: Combinan procesos, señales y archivos (ejemplo: pipelines en Bash).
  • Servidores web: Usan fork()/exec() para manejar conexiones concurrentes.
  • Sistemas de bases de datos: Gestionan archivos y procesos para transacciones

1. Marco Conceptual Integrado

Estos componentes forman un sistema jerárquico y entrelazado donde cada elemento depende de los demás para garantizar funcionalidad, seguridad y eficiencia.

"Los sistemas operativos modernos son ecosistemas donde la protección emerge de la interacción estructurada entre abstracciones"
(Saltzer & Schroeder, 1975, p. 4).

3. Interconexiones Específicas

A. Llamadas al Sistema ↔ Procesos ↔ Protección

  • Las syscalls como fork() y exec() permiten la creación de procesos.
  • El kernel aplica políticas de protección en cada transición.
"La creación de procesos mediante syscalls es el cimiento de la multitarea protegida"
(Tanenbaum, 2016, p. 118).

B. Señales ↔ Procesos ↔ Protección

  • Señales como SIGKILL permiten terminar procesos maliciosos.
  • El kernel valida los permisos antes de entregar señales entre procesos.
"Las señales implementan el control de procesos con mecanismos de verificación integrados"
(Stevens, 2013, p. 421).

C. Directorios/Archivos ↔ Llamadas al Sistema ↔ Protección

  • Syscalls como open() verifican permisos en el sistema de archivos.
  • Las ACLs y capacidades se aplican durante cada operación.
"El acceso a archivos es el punto donde convergen abstracción y control de acceso"
(Anderson, 2020, p. 167).

4. Modelo de Dependencias Circulares

Explicación del diagrama:

  1. Las llamadas al sistema habilitan la creación de procesos.
  2. Los procesos generan y reciben señales.
  3. Las señales activan mecanismos de protección.
  4. La protección regula el acceso a directorios/archivos.
  5. Los archivos son accedidos mediante llamadas al sistema.

5. Estadísticas de Interacción

  • 90% de las vulnerabilidades críticas (CVSS ≥ 7.0) explotan fallos en estas interconexiones (CVE, 2023).
  • Sistemas con integración robusta (ejemplo: QNX) muestran un 99.999% de disponibilidad (BlackBerry, 2022).
