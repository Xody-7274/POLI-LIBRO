5.1 Principios del Hardware de E/S

Componentes fundamentales para la interacción con periféricos

Hasta el momento se ha hablado acerca de dos de los cuales podemos considerar son los principales componentes de una computadora: el procesador y la memoria, así como las funciones que le aporta al sistema operativo (abstracciones a los procesos y a la memoria, hilos, espacios de direcciones y archivo, etc.), en esta unidad hablaremos de otro componente que es importante para la interacción entre el sistema operativo y el usuario.

(Tanenbaum & Bos, 2015)

Si bien los dispositivos de entrada y salida pueden ser vistos de manera distinta en los diversos ámbitos, desde ser vistos como chips, alambres, motores, etc., para los ingenieros eléctricos, también son vistos de otra manera por los ingenieros en programación, siendo éstos los que a menudo trabajan con la interfaz en relación con el software y el hardware.

(Stallings, 2018)

Una función del sistema operativo es la de llevar un control sobre todos los dispositivos de entrada y salida que están presentes en un equipo de cómputo, esto puede ser gestionado mediante comandos a dichos dispositivos, detectando las interrupciones y controlando los errores.

(Silva, 2015; Silberschatz, Galvin, & Gagne, 2018)

5.1.1
Dispositivos de E/S

Los dispositivos de entrada y salida (E/S) pueden dividirse principalmente en dos categorías:

Dispositivos de Bloque

Almacenan información en bloques de tamaño fijo, cada uno con su propia dirección. Los tamaños de bloque comunes varían desde 512 bytes hasta 32,768 bytes. Todas las transferencias se realizan en unidades de uno o más bloques completos (consecutivos).

Algunos ejemplos de dispositivos de bloque más comunes son: discos duros, CD-ROMs y memorias USB.

(Tanenbaum & Bos, 2015)

Dispositivos de Carácter

Envían o almacenan un flujo de caracteres, sin importar la estructura de un bloque. No son direccionables y no tienen ninguna operación de búsqueda.

Algunos ejemplos de dispositivos de carácter más comunes son: impresoras, interfaces de red y ratones.

(Tanenbaum & Bos, 2015)

Algo importante que se debe señalar es que cada dispositivo de E/S cubre un rango de velocidad de transferencia de datos, lo que impone una presión en el sistema operativo para obtener un buen desempeño sobre muchos órdenes de magnitud en las velocidades de transferencia de datos. A continuación, se muestran las velocidades de transferencia de datos de algunos dispositivos comunes, como se indica en la Tabla 1.

(Tanenbaum & Bos, 2015)

5.1.2
Controladores de Dispositivos

El sistema operativo debe ser capaz de gestionar y controlar los dispositivos de entrada y salida presentes en el equipo de cómputo. Esta tarea puede resultar compleja debido a diversos factores internos y externos que influyen en su funcionamiento.

Los factores internos incluyen la velocidad de transferencia de datos y las interrupciones enviadas al sistema para el funcionamiento de los dispositivos. Por otro lado, los factores externos se refieren a interrupciones generadas por el usuario al conectar o desconectar dispositivos. Ante estas situaciones, el sistema operativo debe ser capaz de detectar automáticamente cualquier dispositivo nuevo, proporcionar el controlador o manejador adecuado, y revertir el procedimiento cuando el usuario desconecte o elimine el dispositivo.

(Tanenbaum & Bos, 2015)

Desde la aparición del USB (Universal Serial Bus), la administración de dispositivos de entrada y salida ha mejorado considerablemente. Los dispositivos actuales presentan gran variedad de características en cuanto a propósito, velocidad, volumen de información procesada, dirección del flujo de información, protocolo de comunicación y disponibilidad.

(Tanenbaum & Bos, 2015)

La comunicación entre un dispositivo de entrada y salida y el sistema operativo se establece a través de un puerto. Estos puertos están compuestos por un componente mecánico y un componente eléctrico o controlador, que constituye una colección de circuitos electrónicos capaces de manejar un puerto, un colector o un dispositivo.

(Silva, 2015; Silberschatz, Galvin, & Gagne, 2018)

5.1.3
Interrupciones

Las interrupciones son un mecanismo fundamental en los sistemas de entrada/salida (E/S), ya que permiten que los dispositivos y el CPU se comuniquen de manera eficiente. Gracias a las interrupciones, el procesador puede responder a eventos externos sin necesidad de verificar constantemente el estado de los dispositivos, mejorando así el rendimiento general del sistema.

(Tanenbaum & Bos, 2015)

Tipos de Interrupciones

Las interrupciones son señales que permiten a los dispositivos de entrada y salida avisar al CPU si ha ocurrido un evento importante. Se dividen en dos categorías principales:

Interrupciones en el Hardware: Generadas por dispositivos físicos externos (como discos duros, tarjetas de red o teclados). Se consideran asíncronas, ya que pueden ocurrir en cualquier momento, sin seguir un orden predecible. Son de distinto orden de prioridad, donde cada dispositivo utiliza una línea de petición de interrupción (IRQ) para evitar conflictos entre dispositivos.

Excepciones (Interrupciones de Software): Generadas directamente por la CPU cuando se detecta una condición especial, como una división entre cero, un acceso no permitido a la memoria o una llamada al sistema. Son síncronas, pues ocurren como resultado directo de la ejecución de instrucciones.

El siguiente diagrama ilustra el flujo general del manejo de interrupciones y la distinción entre interrupciones de hardware y excepciones, tal como se muestra en la Figura 1.

Figura 1. Manejo de interrupciones de hardware y excepciones software

Las interrupciones permiten que el CPU se entere de eventos importantes sin necesidad de hacer revisiones constantes (técnica conocida como polling). Este mecanismo mejora la eficiencia del sistema operativo en la gestión de dispositivos E/S.

(Silberschatz, Galvin, & Gagne, 2018)

El Proceso de Solución para Interrupciones

El manejo de interrupciones en un sistema operativo sigue una secuencia bien definida de etapas que permiten al CPU responder a eventos externos sin comprometer la estabilidad del sistema. Este proceso asegura que se conserve el estado del programa actual, se atienda adecuadamente la interrupción, y se retome posteriormente la ejecución. Las etapas clave se presentan en la Tabla 2.

Paso Descripción Tiempo/Ciclos
1 Guardar registros, PC 10-20 aprox.
2 Identificar Vector (ISR) 5-10 aprox.
3 Ejecutar ISR Variable
4 Restaurar 10-20 aprox.
Tabla 2. Etapas del proceso de atención a interrupciones

Descripción del Proceso

  1. El CPU detiene momentáneamente la ejecución normal y guarda su estado (registros, PC, etc.).
  2. Identifica el origen de la interrupción consultando el vector de interrupciones.
  3. Ejecuta la Rutina de Servicio de Interrupción (ISR).
  4. Restaura el estado anterior y continúa con la ejecución normal del programa.

Gracias a este mecanismo, el sistema puede atender múltiples dispositivos sin pérdida de información o inestabilidad.

Gestión del Hardware de Interrupciones

Existen distintos métodos para gestionar el hardware de interrupciones. Algunos de los más conocidos son:

Daisy-Chaining: Encadenamiento de señales en serie que utiliza una línea de interrupción compartida (IRQ) y una señal INTA para reconocer interrupciones. Tiene prioridad fija, pero es poco eficiente y no escalable.

APIC (Advanced Programmable Interrupt Controller): Controlador avanzado que permite una gestión más flexible y eficiente de las interrupciones, ya que es programable, dinámico y puede operar en sistemas con múltiples núcleos.

Estos métodos se comparan en la Tabla 3.

Conceptos Daisy Chaining APIC
Modo de conexión En cadena / serie Distribuido con controladores separados
Prioridad Fija, según la cadena Programable y dinámica
N.º de procesadores Para un solo CPU Para múltiples CPU / núcleos

Ventajas de usar Interrupciones

  • Mayor aprovechamiento del CPU
  • Respuesta rápida ante eventos externos
  • Soporte a la multiprogramación

Aplicación práctica de las interrupciones en Software y Hardware

Las interrupciones no solo son un concepto teórico, sino que tienen aplicaciones muy prácticas tanto en sistemas operativos como en programación a bajo nivel.

En el contexto de los sistemas operativos, una forma común de generar interrupciones de software es mediante llamadas al sistema (system calls). Estas permiten que los programas de usuario soliciten servicios del núcleo del sistema, como terminar un proceso, leer un archivo, o acceder a recursos restringidos. En arquitecturas como Linux x86, esto se logra a través de instrucciones ensamblador que colocan un número de servicio en un registro, junto con los parámetros necesarios, y luego se genera una interrupción con int 0x80, que transfiere el control al sistema operativo. Este proceso se puede observar tal como lo muestra el Código 1.

// Ejemplo de llamada al sistema en Linux x86
void syscall() {
    asm("mov $1, %eax\n"   // Número de llamada (exit)
        "mov $0, %ebx\n"   // Código de salida
        "int $0x80");      // Interrupción de software
}
Código 1: Llamada al sistema exit en Linux x86 mediante interrupción software

Por otro lado, en los sistemas embebidos como los microcontroladores AVR, se utilizan interrupciones de hardware para reaccionar ante eventos externos como un temporizador que se desborda, una señal en un pin de entrada, o una recepción por comunicación serial. La respuesta a la interrupción se realiza escribiendo una función conocida como Rutina de Servicio de Interrupción (ISR), que se activa automáticamente cuando ocurre el evento. Un ejemplo de esto se puede ver tal como lo muestra el Código 2, donde un temporizador activa una ISR que conmuta el estado de un LED conectado al pin PB0.

// Código 2 – ISR para interrupción de timer en AVR
#include <avr/interrupt.h>

ISR(TIMER1_COMPA_vect) {
    PORTB ^= (1 << PB0); // Conmuta LED en PB0
}
Código 2. Rutina de Servicio de Interrupción (ISR) para temporizador en microcontrolador AVR

Estos ejemplos demuestran cómo el mecanismo de interrupciones se implementa tanto a nivel de sistema operativo como en hardware de bajo nivel, permitiendo una respuesta eficiente y oportuna ante eventos, y liberando al procesador de realizar tareas de vigilancia constante como el polling.

5.1.4
Acceso de Memoria Directo (DMA)

El DMA (acceso directo a memoria) es un circuito que permite que algunos dispositivos puedan leer o escribir datos en la memoria principal sin necesidad de que el CPU intervenga en cada byte transferido, lo cual contribuye a que las transferencias sean más rápidas. Además, mejora el rendimiento porque libera al procesador de tareas pesadas de entrada o salida.

(Stallings, 2020)

Componentes y Funcionamiento

El DMA opera como un controlador que se encarga de gestionar las transferencias de datos. Los elementos más importantes se presentan en la siguiente Tabla 4:

Componente Función
Registro de dirección Almacena la dirección física en la memoria donde comienza la transferencia
Contador de bytes Indica el tamaño del bloque de datos a transferir
Registro de control Configura el modo de operación (si es de lectura/escritura) y prioridad del dispositivo
Tabla 4. Componentes principales del controlador DMA

El controlador DMA administra las solicitudes de transferencia mediante distintos registros donde se configura:

  • La dirección de memoria donde se van a escribir o leer los datos
  • El número de bytes (cantidad de datos a transferir)
  • El modo de operación, que puede ser de dispositivo a memoria o de memoria a dispositivo

(Tanenbaum & Bos, 2015)

Pasos del Controlador DMA

  1. Configuración inicial: El procesador programa el controlador DMA estableciendo los registros de dirección, el contador de bytes y el modo de operación. Esta configuración se realiza una sola vez por cada transferencia, lo que reduce la carga sobre el CPU.

(Stallings, 2020)

  1. Solicitud de transferencia: Una vez configurado, el periférico puede solicitar la transferencia de datos al estar listo (por ejemplo, tras leer un bloque del disco), o la CPU puede iniciarla si es una operación programada. El DMA valida la solicitud y verifica que el canal esté libre.

(Stallings, 2020)

  1. Toma de control del bus: El controlador DMA solicita el uso del bus del sistema. Este bus es el canal de comunicación entre CPU, memoria y dispositivos. Cuando la CPU libera el bus, el DMA accede a la memoria para ejecutar la transferencia.

(Stallings, 2020)

  1. Transferencia directa: Se realiza la transferencia de datos directamente entre el dispositivo y la memoria. Según el modo de operación, puede realizarse en bloques o segmentos más pequeños. La CPU no participa byte por byte, lo que mejora la eficiencia.

(Stallings, 2020)

  1. Notificación de finalización: Al finalizar la transferencia, el DMA envía una señal de interrupción al procesador, permitiendo que el sistema operativo o el proceso solicitante continúe su ejecución.

(Stallings, 2020)

Ventajas y Desafíos del Uso de DMA

El uso de DMA proporciona múltiples beneficios en el manejo de operaciones de entrada y salida, especialmente en términos de velocidad y eficiencia del procesador. Sin embargo, también introduce ciertos retos técnicos. A continuación, se presenta un resumen de las principales ventajas y desafíos asociados a esta técnica, tal como lo muestra la Tabla 5:

Ventajas Desafíos
Libera a la CPU de la supervisión constante Puede generar contención en el bus
Permite transferencias de datos más rápidas Requiere hardware adicional (controlador)
Reduce la latencia en operaciones de E/S Complejidad en la gestión de prioridades
Tabla 5. Ventajas y desafíos del uso de DMA

Ejemplo de Código de Transferencia con DMA

Para ilustrar cómo se lleva a cabo una transferencia mediante DMA en un sistema computacional, a continuación, el Código 3, presenta un fragmento de código que describe un flujo básico de configuración, ejecución y finalización de la transferencia:

Código 3: Ejemplo de flujo de transferencia con DMA
// Paso 1: Configuración inicial con verificación
if (!dma_controller.set_source_address(origen) ||
    !dma_controller.set_destination_address(destino) ||
    !dma_controller.set_transfer_size(tamaño) ||
    !dma_controller.set_mode(MODO_BLOQUE)) {
    // Manejar error de configuración
    return ERROR_CONFIGURACION;
}

// Registrar manejador de interrupción
registrar_interrupcion(IRQ_DMA, manejador_interrupcion_dma);

// Paso 2: Solicitud de transferencia
if (!dispositivo.solicitar_transferencia()) {
    // Manejar error del dispositivo
    return ERROR_DISPOSITIVO;
}

// Paso 3: Iniciar transferencia
dma_controller.iniciar_transferencia();

// [El sistema principal puede continuar con otras tareas]

// Paso 5: Manejador de interrupción
volatile bool transferencia_completa = false;
volatile int estado_transferencia = 0;

void manejador_interrupcion_dma() {
    if (dma_controller.verificar_error()) {
        estado_transferencia = ERROR_TRANSFERENCIA;
    } else {
        estado_transferencia = EXITO;
    }
    transferencia_completa = true;
    // Notificar al sistema/scheduler/otro proceso
    notificar_sistema(EVENTO_DMA_COMPLETO);
}
